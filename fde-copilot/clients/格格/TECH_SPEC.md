## 交付形态（倾向，待客户确认 Q10）
默认按**单用户 Web 应用（个人求职作战台）** 落地：格格一人使用，本地/私有部署即可，无多租户、无复杂权限。若客户更想要"一批可交付材料 + 轻脚本"而非软件，则退化为文档生成流水线——架构见下，二者共用同一套生成内核。

## 架构（分层）
- **前端**：轻量 Web UI（仪表盘：画像/岗位看板/资产库/演练）。单用户，无需登录体系（本地或口令保护即可）。
- **应用层**：
  - 画像与差距引擎（F1）
  - 岗位情报采集与匹配（F2）——定时抓取 + 规则/LLM 匹配
  - 资产生成服务（F3 简历 / F4 教案 / F6 试讲稿）——LLM 驱动的模板化生成
  - 投递看板与提醒（F5）
  - 模拟评委对练（F6）——LLM 多轮对话
- **数据层**：本地数据库（SQLite/文件）足够；存格格档案、岗位库、生成资产、投递状态。

## 数据模型（核心实体）
- `Profile`：格格档案（学历/在读状态、研究方向、成果、语言、党员、年龄、目标地域、意向学院）。
- `Job`：岗位（院校、岗名、学历/经验要求、是否接受海外博士、截止日、来源URL、匹配标签+理由）。
- `Asset`：生成资产（类型=简历/教案/课程设计/试讲稿，关联Job可选，版本，评审分数）。
- `Application`：投递记录（关联Job，状态机：待投→已投→初筛→试讲→面试→offer/拒，提醒时间）。
- `GapReport`：差距报告（满足项/缺口项/补齐动作/优先级/就绪度分）。

## 接口/依赖
- **LLM 接入**：需大模型做生成与匹配。用户环境已有 **HiLinkup Gateway**（OpenAI 兼容，可调 GLM/Kimi/gpt 等）——倾向作为统一模型接入层（**key 位置待确认**，见 MEMORY）。境内使用倾向国产模型合规。**待客户确认是否采用（GAPS G4）。**
- **岗位数据源**：高校人才网/科学人才网/各校人事处公告等——需确认可合规抓取方式；抓不到的字段（如"是否接受泰博"）标"待确认"，不臆造。
- **文档导出**：教案/简历导出为可编辑 docx/pdf。

## 部署
- 单用户、私有/本地部署优先；无需高可用。数据含格格个人信息，注意本地存储与最小外发。

## 技术风险
- **岗位抓取的合规与稳定性**：站点结构变动/反爬——需容错与人工兜底录入。
- **LLM 生成质量与真实性**：教案/简历须防"看起来像但不能用"；一律"初稿+格格审校"，杜绝虚构经历。
- **模型接入 key/合规未定**：HiLinkup Gateway 采用与否、key 位置未确认——阻塞生成类功能联调。
- **"是否软件系统"未最终拍板**（Q10）：影响前端投入规模，但生成内核不受影响，风险可控。
