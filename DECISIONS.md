## D1 · 发布能力的多租户处理（2026-07-15，细节待续）

**问题**：小红书 / Twitter 这类**会话型平台**（无官方发帖 API、靠登录 Cookie / 浏览器 session），能不能做成我们托管的多租户发布服务？

**难点**：不在技术隔离（一台机器技术上能跑多个隔离的浏览器上下文），而在**平台风控**——同一台机器 / 同一 IP / 同一指纹挂多个账号，会被判为机器人矩阵，**限流甚至封号**；且 Cookie 会过期、要反复重登。

**结论**：

1. **会话型平台的「发布」不做成中心化托管多租户** → 让用户**自部署**，**发布在自己的电脑上**（用自己的 IP、自己的账号）。本地发布甚至可用小模型。
2. **「生成」类可以由我们提供 / 中心化**：写文章 / 生成文章、生成插图、capability-packs 的各种能力、**搜索（agent-reach）**——这些交给我们没问题。
3. **拆分原则**：**generation（生成）= 可中心 / 可托管；publish（会话型发布）= 边缘 / 用户自部署。**
4. **API 型平台（公众号：AppID/Secret、无状态调用）** 可以中心化多租户；会话型才是需要推到边缘的那类。

**落地方向**（待细化）：capability-packs 账号结构做成**每平台多账号** + 按「API 型 / 会话型」**分类路由**；会话型走用户自己的边缘实例（单用户多号用独立浏览器上下文隔离）。

**状态**：方向已定，实现细节「后边再聊」。
